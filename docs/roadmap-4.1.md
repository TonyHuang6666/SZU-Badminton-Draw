# 4.1 开发计划

4.1 的目标不是继续堆按钮，而是把 4.0 已经跑通的“单项目赛程 + 多项目工作台 + 合并材料包”打磨成更接近裁判长真实工作台的版本。

## 1. 高级赛程约束

优先做提醒，再逐步做自动安排。

- 已开始：单项目赛程生成后会按“校园赛/正式赛”档案生成高级约束提醒，不自动改变赛程。
- 决赛或重点比赛指定场地、指定时间段。
- 半决赛、决赛自动增加休息间隔。
- 重点比赛错峰，避免观众和裁判资源被分散。
- 选手不可用时间、场地不可用时间。
- 裁判员、记录员、志愿者数量约束。
- 休息间隔配置档：校园赛默认短间隔，正式赛可切到更严格的间隔。

## 2. 团体赛阵容与记分表

团体赛要尊重正式规则，也要保留校园赛现场调整空间。

- 支持三场制、五场制的项目顺序配置。
- 导入或填写每轮团体赛上场名单。
- 检查同一运动员同一轮团体赛是否在同一项目重复出场。
- 允许裁判长根据兼项情况调整团体赛场序。
- 团体记分表继续使用 Excel，便于现场填写和二次打印。

## 3. 多项目材料包增强

- 已完成：多项目“自动调整冲突”升级为全局自动编排未完成赛程，锁定已完成场次，统一检查淘汰依赖、场地占用、兼项选手时间冲突和最小休息间隔。
- 支持选择导出某几个比赛日，而不是每次导出全部比赛日。
- 合并材料包增加打印清单，列明每个文件用途和推荐打印份数。
- 合并赛程记录表导入回各项目 `.szbd` 的安全流程。
- 自动识别已经导出的材料包版本，避免覆盖现场正在填写的表。

## 4. WPF 与 Avalonia 收敛

- Avalonia 作为主力桌面版继续开发。
- WPF 只做关键 bug 修复和发布兜底。
- 新功能必须先落在 `Core`、`Excel`、`Workflows`，再接 UI。
- 抽离单项目和多项目时间场地表格的共享视图模型，减少两套界面重复逻辑。

4.1 的新增交互优先接 Avalonia；WPF 如无现场阻塞问题，不再同步每个新控件。

## 5. macOS 正式发布

- Developer ID 签名。
- Notarization 公证。
- Staple。
- Gatekeeper 验证。
- 首次启动提示优化。

## 6. 循环赛统计

- 记录表导入后自动计算循环赛胜场、净胜局、净胜分。
- 支持同分时按规则排序。
- 生成循环赛名次审计表，方便裁判长复核。

## 开发顺序

每项功能仍按以下工程落地顺序推进：

```text
Core（规则） -> Excel（文件读写） -> Workflows（统一流程） -> Avalonia -> WPF 关键兜底 -> xUnit 测试 -> 文档
```

如果 WPF 进入维护期，WPF 关键兜底只包含会影响 Windows 用户继续办赛的流程，不再要求每个新交互都同步完整实现。
