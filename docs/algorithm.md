# 抽签算法说明

本文说明 2.0 版本的核心算法。更详细的规则条文对应关系见 [rules-compliance.md](rules-compliance.md)。

## 背景

原始 `BFSZU.cpp` 使用控制台输入名单，通过随机编号、均匀分组、种子分散和首轮轮空名单计算，帮助深大羽协快速完成比赛抽签。原始仓库见 [TonyHuang6666/Badminton-Draw](https://github.com/TonyHuang6666/Badminton-Draw)。

新版保留“快速、透明、适合干事操作”的历史目标，但把旧的经验式流程整理成可审计的核心算法。GUI 中不再保留旧 v1 算法选择，避免同一场比赛因为误选算法而产生不可比对的结果。

## 数据模型

每个参赛对象会被转成一个 `DrawParticipant`：

- 单打：一个选手。
- 双打：一对选手，显示为 `[选手A 选手B]`。
- 团体：一个队伍，通常来自名单模板的“学院/学部”列。
- 种子：通过 `IsSeed` 和 `SeedRank` 标记。
- 学院/学部和搭档学院/学部：作为同单位回避信息保留。

核心算法只处理参赛对象、种子序号、同单位信息和设置项，不直接依赖 Excel 或 WPF 界面。

## 可复现随机

程序使用公开的随机数种子驱动稳定伪随机过程。同一份名单、同一组设置、同一个随机数种子和同一程序版本，会得到相同结果。

输入哈希覆盖：

- 比赛模式
- 项目类型
- 小组数
- 算法版本
- 淘汰赛目标
- 名单中的显示名称、种子信息、学院/学部信息

这让复查者可以确认“复现时用的是不是同一份输入”。

## 通用分组

通用分组发生在淘汰赛和循环赛之前：

1. 计算每个小组的目标人数，各组人数差不超过 1。
2. 将种子按 `SeedRank` 从高到低放入受保护的小组顺序，例如 4 组时优先落在第 1、4、2、3 组。
3. 将非种子按随机数种子稳定洗牌。
4. 每放入一个参赛对象时，优先选择未满员、同单位冲突更少的小组。
5. 对每个小组内部再次稳定洗牌。

这个流程对应规则中“分组循环赛可抽签分组”和“同队运动员均匀分布”的思想。对于深大校内赛，“同队”在程序中落到“学院/学部”“搭档学院/学部”等单位字段。

## 种子保护

程序不负责判断谁应该成为种子。种子应由赛事组织方依据近期成绩、技术水平或校内规程确定。程序负责三件事：

1. 校验种子数量不能超过规则分档上限。
2. 校验种子序号为正整数且不能重复。
3. 在分组和淘汰树中执行种子保护。

种子数量分档如下：

| 参赛数量 | 最多种子数 |
| ---: | ---: |
| 少于 16 | 2 |
| 16 至 31 | 4 |
| 32 至 63 | 8 |
| 64 至 128 | 16 |
| 超过 128 | 32 |

淘汰赛签位使用官方保护位置表。示例：

| 签位规模 | 种子位置顺序 |
| ---: | --- |
| 16 | 1、16、5、12 |
| 32 | 1、32、9、24、5、13、20、28 |
| 64 | 1、64、17、48、9、25、40、56、5、13、21、29、36、44、52、60 |

对于 128 和 256 签位，程序同样内置对应保护位置。对于 2、4、8 这类小签表，程序使用顶部/底部的远离原则。

## 淘汰赛首轮赛和轮空

规则要求参赛数不是 2 的乘方时，第一轮设置轮空。程序对每个小组独立计算：

```text
groupSize = 小组人数
base = 不超过 groupSize 的最大 2 的乘方
首轮赛人数 = 2 * (groupSize - base)
轮空或直接进入正赛人数 = groupSize - 首轮赛人数
```

例子：

| 小组人数 | base | 首轮赛人数 | 轮空或直接进入正赛人数 |
| ---: | ---: | ---: | ---: |
| 3 | 2 | 2 | 1 |
| 4 | 4 | 0 | 4 |
| 5 | 4 | 2 | 3 |
| 6 | 4 | 4 | 2 |
| 7 | 4 | 6 | 1 |
| 8 | 8 | 0 | 8 |

切分首轮赛时，程序会优先把种子放入轮空或直接进入正赛的位置。只有当某个小组的种子数量超过可用轮空名额时，多出来的种子才会进入左侧首轮赛。

## 淘汰赛目标

淘汰赛通过 `KnockoutGoal` 明确目标：

- `OneQualifierPerGroup`：每组出线。每个小组独立打到 1 个出线名额，最终不再跨组决冠军。
- `Champion`：决出冠军。用于最终需要产生冠军的赛制。

当小组数不是 2 的 n 次幂时，程序会把 `Champion` 规范化为 `OneQualifierPerGroup`。原因是非 2 的 n 次幂小组数无法自然接入后续完整淘汰段，强行生成会出现不合理的 `7进3`、`3进1` 等轮次。

当小组数是 2 的 n 次幂且大于 1 时，程序允许两种目标：

- 选择 `每组出线`：例如 29 队分 8 组，最终得到 8 个小组出线名额。
- 选择 `决出冠军`：先让 8 个小组各出线 1 队，再接入 8 强、4 强、决赛，最终产生冠军。

这是深大校内赛的关键变通：粤海、丽湖或不同学院片区预选时，小组数常常反映“出线名额分配”，而不是最终冠军签表规模。

## 循环赛

循环赛会导出矩阵式 `对阵表`。个人赛循环赛和团体赛循环赛本质相同，只是参赛对象名称不同：个人赛显示选手或组合名，团体赛显示队伍或学院名。

程序行为：

- 对角线为空轮。
- 右侧保留胜场、净胜、名次列。
- 矩阵中标出场次编号。
- 每个小组下方生成赛程顺序。
- 个人赛循环赛如存在同学院/同单位对阵，会把这些比赛优先排到前面。

团体赛循环赛不显示“同单位优先”提示，因为团体赛中每个参赛对象本身就是一个单位。

## 对阵表连接线

导出的淘汰赛 `对阵表` 使用空白单元格边框绘制连接线。3 人组、5 人组这类有首轮赛的小组会先把首轮赛胜者接入正赛位置，再与轮空或直接晋级者连接到下一轮。

当不同小组人数不一致时，表格列数仍按最大组内签表统一铺开，但第 4 行表头按整张表的总签位数汇总，避免把“单组内轮次”误读成“整项赛事轮次”。例如 29 队分 8 组时，既有 3 队组也有 4 队组，首轮赛后共有 26 个正赛签位，表头会显示 `26进13`、`13进8`、`出线`。

## 当前边界

- 程序不自动根据历史成绩计算种子，只执行组织方填入的种子。
- 程序不自动处理抽签后的退赛、替补和重抽。
- 循环赛表格保留统计列，但当前不自动判定最终名次。
- 如果赛事要求同批次种子在保护区内再次抽签，应先公开完成同批次排序或未来扩展为显式二次抽签功能。
