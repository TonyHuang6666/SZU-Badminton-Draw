# WPF 与 Avalonia 取舍评估

## 当前判断

4.0 起，Avalonia 版已经覆盖抽签、单项目赛程、赛事存档、记录表导入、单场计分表、多项目赛程编排、兼项明细、拖拽调整、合并材料包和跨平台中文 PDF 输出。它应作为后续主力桌面版。

WPF 版仍有存在意义，但定位应从“主版本”调整为“Windows 备用版本”和“短期回归对照”。

## 为什么不立刻删除 WPF

- Windows 现场机器环境不可控，WPF 是一个成熟兜底。
- 4.0 刚完成跨平台主链路，保留 WPF 有助于判断 Avalonia 问题是 UI 层还是共享业务层。
- 已经发布了 WPF 4.0 单文件程序，短期内删除会增加用户迁移成本。

## 为什么新功能不应优先写 WPF

- 双 UI 同步会快速消耗开发精力。
- 多项目工作台、时间场地表格和独立窗口已经更依赖 Avalonia 的跨平台布局能力。
- Windows 和 macOS 共用 Avalonia 后，真实用户反馈可以集中到一套界面。

## 建议策略

- 4.1：Avalonia 优先；WPF 只修严重 bug。
- 4.2：如果 Avalonia Windows 版稳定，WPF 标记为维护模式。
- 5.0：若没有 WPF 独有用户需求，可以移除 WPF 发布资产，保留源码一段时间作为历史参考。

## WPF 退场条件

- Avalonia Windows 版连续至少一个正式版本完成真实办赛流程验证。
- Avalonia 能稳定导出 Excel、PDF、图片和材料包。
- 多项目赛程工作台在 Windows 上无严重卡死或文件膨胀问题。
- 没有 WPF 独有功能仍未迁移。
- README 和 release 说明明确告知用户默认下载 Avalonia 版。

## 维护边界

WPF 仍应能：

- 打开项目。
- 完成基础抽签。
- 完成既有赛程和材料包流程。
- 通过共享测试保证 `Core`、`Excel`、`Workflows` 行为不漂移。

WPF 不再优先承接：

- 新的复杂视图交互。
- 多窗口缩放体验优化。
- 新增跨平台视觉细节。
- 只影响 macOS 的包体和启动体验问题。
